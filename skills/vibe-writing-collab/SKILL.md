# Vibe Writing 共创工作流

## 🎯 Skill 介绍

**Vibe Writing** 是一套完整的人AI共创写作系统，帮助你快速将想法、经历、灵感转化为高质量的文章、分享稿、演讲稿或内部文档。

### 💡 核心价值

- **保留你的原声**：AI 不替代你的表达，只是帮助你更清晰、更有力地传达
- **高效的协作流程**：从素材→结构→初稿→迭代→定稿，每一步都对齐，避免返工
- **系统化管理**：所有素材、讨论、版本都有记录，随时可查阅、回退、复用
- **自动化初始化**：一键创建项目，自动生成工作模板，开箱即用

### 🎨 适用场景

| 场景 | 特点 | 效果 |
|------|------|------|
| 📝 **博客文章** | 有想法但不知如何组织 | 快速成稿，逻辑清晰 |
| 🎤 **分享演讲稿** | 口述内容需要打磨 | 保留亲切感，结构完整 |
| 📢 **内部文档** | 需要反复迭代调整 | 版本管理清晰，沟通高效 |
| 💭 **经验总结** | 经历丰富但难以表述 | 洞察深刻，表达精准 |
| 🚀 **项目文案** | 多人协作，需求多变 | 讨论记录完整，决策可追溯 |

### 🚀 快速开始

1. **说出你的想法** → "我想写一篇分享"
2. **AI 收集素材** → 边讲边记，随时追问洞察
3. **梳理结构** → AI 提出逻辑框架，确认后开写
4. **迭代打磨** → 反复修改，精益求精
5. **输出成果** → 支持多种格式（HTML/Word/PDF）

## 📖 详细工作流

以人为主体，AI 作为协作伙伴。用户提供原始素材（经历、想法、口述），AI 负责梳理、分析、润色、写稿。

## 工作流（5个阶段）

### Phase 0：项目初始化

用户启动时：

1. **确认关键信息**（可一次性问完）：

   - 目标读者是谁？
   - 分享场景（时长/形式/平台）？
   - 核心想传达什么？
   - 有没有参考样本或风格偏好？

2. **建立项目目录**：

   ```
   <current-project-root>/项目/[项目名]/
   ├── 聊天记录与素材.md    # ✨ 自动生成（基于参考模板）
   ├── 正文初稿-v1.md
   └── 正文初稿-vN.md
   ```

   **路径规则**：
   - 自动识别用户当前项目的根目录
   - 在根目录下创建 `项目/[项目名]/` 目录
   - 若 `项目/` 目录已存在则复用，若项目子目录已存在则提示并确认是否覆盖

3. **自动生成 `聊天记录与素材.md`**：

   基于参考模板自动生成工作文件，并进行以下替换：
   - `{{PROJECT_NAME}}` → 用户输入的项目名称
   - `{{DESCRIPTION}}` → 用户输入的项目描述
   - `{{DATE}}` → 当前日期（YYYY-MM-DD）

   **自动生成的文件包含**：
   - **项目基本信息**：项目名称、创建日期、项目描述
   - **核心信息**：目标读者、分享场景、核心想法、参考样本
   - **素材收集区**：素材编号、标题、来源、记录时间、原始素材内容（保留原有语气）、核心洞察（1句话提炼）、相关标签
   - **讨论记录**：时间、参与者、讨论要点、决策、后续行动
   - **参考资源**：文档链接、样本代码、设计稿、竞品分析
   - **进度跟踪**：各阶段完成度表
   
4. 创建素材文件后告知用户："已建好工作目录，开始聊吧，随时记录。"

---

### Phase 1：素材收集（核心阶段）

**模式**：用户口述/描述，AI 随时记录 + 追问。

每当用户分享一段素材：

1. **追问提炼**：问 1-2 个追问，挖掘更深的洞察（如：「这件事里最让你惊喜的是什么？」「如果用一句话总结这个体验？」）
2. **实时写入**素材文件，格式：
   ```markdown
   ### 素材N：[自动命名标题]
   **来源：用户口述，YYYY-MM-DD**
   > [用户原话或核心内容，保留原有语气]
   **核心洞察**：[1句话提炼]
   ---
   ```
3. **不要打断节奏**：用户连续输出时，先记录，再追问。

素材收集完毕的信号：

- 用户说「差不多了」「开始写吧」
- 或素材已覆盖核心主题

---

### Phase 2：结构规划（动手前对齐）

素材收集完毕后，**主动输出结构方案**，不要直接开始写稿。

输出格式：

```
【建议结构】
一、[章节名] — [1句描述逻辑]
二、[章节名] — [1句描述逻辑]
...

【核心线索】[这篇文章/分享的主干论点]
【亮点素材】[哪2-3个素材最有力]
【建议删减】[哪些素材可以不用]
```

等用户确认或调整结构后，再进入写稿。

---

### Phase 3：迭代写稿

**原则**：

- 每次完整输出一个版本，存为 `正文初稿-vN.md`
- 用户的原话/感悟尽量保留原文，只适当润色语气，不改变原意
- 有分享时长要求时，控制篇幅（约 200字/分钟）

**版本管理**：

- v1：按确认的结构生成完整初稿
- v2+：根据用户反馈迭代（明确说明每版改了什么）

**用户反馈处理**：

- 「这段不对」→ 问清楚哪里不对，精准修改，不要大改
- 「感觉差一点」→ 追问「哪方面差？语气/逻辑/内容？」
- 「删掉这段」→ 执行后确认是否需要补充其他内容衔接

---

### Phase 4：最终确认 & 输出

文字版确认 OK 后，询问是否需要：

- [ ] 生成 HTML 分享页面
- [ ] 发布到学城文档（需 `citadel skill`或者 `km-user` mcp）
- [ ] 导出 Word/PDF

**HTML 页面**：仅在用户明确确认文字版后才构建，不要提前生成。

---

## 关键行为规范

**随时记录**：只要用户说了有价值的内容，就写入素材文件，不要等用户说「记一下」。

**简洁追问**：每次最多问 2 个问题，不要一次列出 10 个问题。

**保留原声**：用户原话里的口语化表达、个人感悟，是分享的灵魂，不要过度「正式化」。

**主动推进**：每个阶段结束后，主动提示下一步（「素材差不多了，我来梳理一下结构？」）。

**版本留档**：每个版本都存文件，不要覆盖，方便用户回退。
