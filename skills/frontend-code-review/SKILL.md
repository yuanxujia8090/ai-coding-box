# Web 前端项目编码规范检查

对前端代码进行系统化的规范检查与审查。

## 规范级别

每条规范分为两个级别：

- **【强制】**：必须遵守，违反时应报告为错误（error）
- **【建议】**：推荐遵守，违反时报告为警告（warning）

## 检查流程

### 1. 确定检查范围

收到用户的代码审查请求后，先判断代码涉及的技术栈：

- 纯 JavaScript → 读取 `references/javascript.md` + `references/coding-style.md`
- TypeScript → 读取 `references/typescript.md` + `references/coding-style.md`
- React (JSX/TSX) → 额外读取 `references/react.md`
- Vue (.vue) → 额外读取 `references/vue.md`
- 单元测试代码 → 额外读取 `references/unit-test.md`
- 文件/目录/包结构问题 → 读取 `references/project-structure.md`

如果不确定涉及哪些技术栈，读取所有 references 文件。

### 2. 逐文件检查

对每个待审查的文件，按以下维度依次检查：

1. **文件与目录规范**：文件命名、目录结构、包管理
2. **编码风格**：缩进、分号、逗号、引号、空格、换行、括号、注释、命名
3. **语言规范**：变量、对象、数组、函数、类、模块、Promise 等语言特性使用
4. **框架规范**：React 组件规范 或 Vue 组件规范（按实际框架）
5. **单元测试规范**：测试用例设计、执行、覆盖率（如涉及测试代码）

### 3. 输出审查报告

使用 `::code-comment` 指令输出每个发现，格式要求：

- title 中标注级别：`[强制]` 或 `[建议]`，以及简短的问题描述
- body 中说明违反了哪条规范、为什么需要修改、如何修改
- 准确标注 file、start、end 行号
- 强制规范 priority=2，建议规范 priority=1

审查完成后，输出一段总结，包含：检查的文件数、发现的问题总数（强制/建议分别多少）、主要问题类别。

### 4. 检查要点速查

以下是各维度最关键的强制规范（完整规范见 references 目录）：

**编码风格核心强制项：**
- 使用 2 个空格缩进，禁止 tab 或 4 空格
- 强制语句末尾使用分号
- 默认使用单引号（JSX/Vue 模板属性用双引号）
- 添加额外结尾逗号（trailing comma）
- 左大括号前空一格，控制语句条件前后空一格
- 单行代码不超过 120 字符
- 变量/函数用小驼峰，类/组件用帕斯卡，常量用宏命名法，文件用连字符命名法

**JavaScript 核心强制项：**
- 使用 const/let，禁止 var
- 一次声明一个变量，禁止链式声明
- 使用字面量创建对象和数组
- 使用对象方法简写和属性简写
- 使用对象展开语法而非 Object.assign
- 使用模板字符串而非字符串拼接
- 使用默认参数语法，禁止修改参数
- 始终使用 === 和 !==
- 禁止嵌套三元表达式

**TypeScript 核心强制项：**
- 必须开启 strict 相关 tsconfig 配置（新项目）
- 可选属性不可赋值 undefined（除非类型显式包含）
- 禁止隐式 any
- 禁止不必要的类型断言和泛型约束
- 类型/接口/枚举用帕斯卡命名法
- 枚举不能有重复值，不能数字字符串混用
- 禁止使用 namespace，使用 ES Module

**React 核心强制项：**
- 无 state 的组件必须用函数组件
- 组件文件用帕斯卡命名，扩展名 .jsx/.tsx
- JSX 属性值用双引号，其他用单引号
- 自闭合标签尾部保留一个空格
- 始终使用 useCallback/useRef 获取 ref，禁止字符串 ref
- Hooks 必须在顶层调用，useEffect 依赖数组必须完整

**Vue 核心强制项：**
- 模板中标签属性必须带双引号
- Prop 声明用小驼峰，模板中用连字符
- 不要省略闭合标签，无子元素用自闭合
- 禁止 v-if 和 v-for 同时作用于一个元素
- 模板中禁止使用 this
- 组件名用帕斯卡命名法
- 使用指令缩写（: 代替 v-bind，@ 代替 v-on）

**单元测试核心强制项：**
- 代码变更合入前和发布前必须确保单测通过
- 使用 Mock 模拟外部依赖，禁止访问数据库或网络
- 测试用例之间不能互相调用或依赖执行顺序
- 禁止使用恒真断言和弱测试断言
- 测试必须幂等，多次运行结果一致
- 不测试内部实现细节和私有属性
