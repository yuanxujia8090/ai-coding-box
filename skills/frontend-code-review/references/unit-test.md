# 单元测试规范

## 一、拟定背景

从保证产品线上质量、提供良好用户体验的角度出发，需要团队建立统一的前端项目单元测试规范，用于保证研发质量，保障线上服务稳定性。

## 二、名词解释

- **前端项目**：代码作用于 Web 应用、小程序应用和 React Native 应用的项目。一般来说，可按代码仓库作为区分项目的单位。
- **单元测试**：指对代码中的最小可测试单元进行检查和验证。在前端领域中，代码的最小可测单元可以是 JavaScript 语言或其子集语言（如 WXS）、超集语言（如 TypeScript、JSX）实现的函数、对象、类或模块。
- **测试用例**：对某个代码可测单元的某一项特性执行的测试任务。针对单次测试执行，测试用例有通过、失败两种状态。
- **代码覆盖率**：描述了程序源代码已经过测试的程度。如无特殊说明，特指基于代码行统计的覆盖率。

## 三、规范内容

### 1. 通用规范

- **【强制】** 项目开发人员引入单元测试时应当确定单元测试的覆盖范围，即希望验证的可测单元具体包含哪些函数、对象、类或模块。
- **【建议】** 单元测试的基本目标：新增代码行覆盖率达到 ≥ 50%，核心模块的行覆盖率和分支覆盖率都要达到 100%。
- **【强制】** 单元测试的覆盖范围不应包含第三方代码，例如 npm 包等。
- **【强制】** 单元测试通过的条件有两个：一是所有测试用例均测试通过，二是代码覆盖率不低于基线。如果有任何测试用例失败，则单元测试失败。如代码覆盖率低于基线，则单元测试失败。
- **【强制】** 项目必须在代码变更合入前确保单元测试通过。
- **【强制】** 项目必须在发布前确保单元测试通过。
- **【建议】** 应尽量扩大项目中单元测试的覆盖范围。建议将项目开发人员能够变更的所有项目代码纳入单元测试的覆盖范围。

### 2. 目录和文件规范

- **【建议】** 目录结构规范：测试文件放在统一目录（如 `tests` 或 `__tests__`）下，或按照功能模块组织。
- **【建议】** 文件命名规范：测试文件名应与源码文件名相对应，后缀为 `.test.js`、`.spec.js`、`.test.ts`、`.spec.ts`。

### 3. 用例设计和开发规范

- **【建议】** 编写单元测试代码遵守 BCDE 原则，以保证被测试模块的交付质量：
  - B（Border）：边界值测试，包括循环边界、特殊取值、特殊时间点、数据顺序等。
  - C（Correct）：正确的输入，并得到预期的结果。
  - D（Design）：与设计文档相结合，来编写单元测试。
  - E（Error）：强制错误信息输入（如非法数据、异常流程、非业务允许输入等），并得到预期的结果。

- **【强制】** 避免过多外部依赖：减少测试对外部模块的依赖，可以使用 Mock 来模拟外部行为。单元测试是可以重复执行的，不能受到外界环境的影响，避免在测试中访问数据库或网络资源。落地方式：CodeReview。

```javascript
// 正面例子：使用 Mock
jest.mock('./api');
test('fetchUser returns user data', async () => {
  api.getUser.mockResolvedValue({ name: 'John' });
  const user = await fetchUser(1);
  expect(user.name).toBe('John');
});

// 反面例子：直接访问网络
test('fetchUser returns user data', async () => {
  const user = await fetch('https://api.example.com/users/1');
  expect(user.name).toBe('John');
});
```

- **【强制】** 核心业务、核心应用、核心模块的增量代码确保单元测试通过，测试代码需要及时更新，以保障测试质量。如果新增代码影响了原有单元测试，请及时修正。
- **【强制】** 对于单元测试，要保证测试粒度足够小，有助于精确定位问题。单测粒度至多是类级别，一般是方法级别。
- **【强制】** 为了保证单元测试稳定可靠且便于维护，单元测试用例之间不能互相调用，也不能依赖执行的先后次序。

```javascript
// 正面例子：独立的测试用例
test('add function', () => {
  expect(add(1, 2)).toBe(3);
});
test('subtract function', () => {
  expect(subtract(3, 1)).toBe(2);
});

// 反面例子：测试用例之间有依赖
let result;
test('add function', () => {
  result = add(1, 2);
  expect(result).toBe(3);
});
test('uses result from previous test', () => {
  expect(subtract(result, 1)).toBe(2); // 依赖上一个测试的结果
});
```

- **【强制】** 单测不允许使用恒真断言，不允许使用弱测试断言。

```javascript
// 正面例子
expect(result).toBe(42);
expect(list).toHaveLength(3);

// 反面例子
expect(true).toBe(true);        // 恒真断言
expect(result).toBeTruthy();     // 弱测试断言（不够精确）
```

- **【强制】** 测试程序应该能运行多次保持结果一致性，保证测试是幂等性。

```javascript
// 正面例子：每次测试前重置状态
beforeEach(() => {
  counter.reset();
});
test('increment counter', () => {
  counter.increment();
  expect(counter.value).toBe(1);
});

// 反面例子：依赖全局状态
test('increment counter', () => {
  counter.increment();
  expect(counter.value).toBe(1); // 第二次运行时会失败
});
```

- **【建议】** 不要有流程控制语句（switch、if 等）。一个好的测试用例处理顺序简单直接：准备数据 → 执行操作 → 验证结果。如有必要，使用子方法分解结构，让测试用例更加易读。

- **【强制】** 不对内部实现细节和私有属性做测试，只关注公开的接口和方法，确保它们满足预期的功能和契约。通过避免测试内部实现细节，可以保持测试的稳定性和可维护性，同时尊重封装性原则。

```javascript
// 正面例子：测试公开接口
test('calculator add', () => {
  const calc = new Calculator();
  expect(calc.add(1, 2)).toBe(3);
});

// 反面例子：测试内部实现
test('calculator internal state', () => {
  const calc = new Calculator();
  calc.add(1, 2);
  expect(calc._internalSum).toBe(3); // 测试私有属性
});
```

### 4. 测试执行规范

- **【强制】** 单元测试应该是全自动执行的，并且非交互式的。测试用例通常是被定期执行的，执行过程必须完全自动化才有意义。输出结果需要人工检查的测试不是一个好的单元测试。单元测试中不准使用 `console.log` 来进行人肉验证，必须使用 `assert` 来验证。

### 5. 测试报告规范

- **【强制】** 测试报告必须包含单元测试是否通过、通过的测试用例的数量和详情、失败的测试用例的数量和详情，以及代码覆盖率。
- **【强制】** 测试报告的失败用例详情中，必须明确指出失败用例的名称，以及预期结果和实际结果的差异。
- **【强制】** 测试报告应包含按目录和文件聚合的代码覆盖率，并标注未覆盖的代码行。
